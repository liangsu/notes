夸库分页：
1. 全局视野法
1.1 将order by time offset X limit Y，改写成order by time offset 0 limit X+Y。
1.2 服务层将改写后的SQL语句发往各个分库：即例子中的各取3页数据。
1.3 假设共分为N个库，服务层将得到N*(X+Y)条数据：即例子中的6页数据。
1.4 服务层对得到的N*(X+Y)条数据进行内存排序，内存排序后再取偏移量X后的Y条记录，就是全局视野所需的一页数据。

2. 业务折衷一：禁止跳页查询
2.1 查第一页的方法和“全局视野法”一样，但点击“下一页”时，需要拉取第二页数据，在第一页数据的基础之上，能够找到第一页数据time的最大值
2.1 将查询order by time offset 100 limit 100，改写成order by time where time>$time_max limit 100

3. 业务折衷二：允许数据精度损失
3.1 要查询全局的第100页（每页50条）的数据，对于单库：offset 9900 limit 100改写为offset 4950 limit 50。对于跨库查询：每个分库偏移4950（一半），获取50条数据（半页），得到的数据集的并集，基本能够认为是全局数据。当然这一页数据的精度并不是精准的。

4. 二次查询法
如要查询 select * from T order by time offset 1000 limit 5，有3个库
4.1 如果是3个分库，则可以改写为 select * from T order by time offset 333 limit 5
4.2 找到所返回3页全部数据的最小值
4.3 查询二次改写 select * from T order by time where time between time_min and time_self_max, time_self_max为4.1 查询出来的最大值
4.4 在每个结果集中虚拟一个time_min记录，找到time_min在全局的offset
4.5 既然得到了time_min在全局的offset，就相当于有了全局视野，根据4.3的结果集，就能够得到全局offset 1000 limit 5的记录


主从数据不一致解决方案：
1. 业务可以接受，不优化
2. 强制读主，高可用主库，用缓存提高读性能
3. 在cache里面记录哪些记录发生过写请求，来路由读主还是读从
